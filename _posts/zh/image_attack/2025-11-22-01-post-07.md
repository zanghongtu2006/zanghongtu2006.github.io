# 第七章 · 工程实践：如何构建一个可扩展的攻击执行器（Attack Runner）

本章从工程角度讲清楚一件事：

> 如何把各种攻击算法（FGSM / PGD / C&W / HSJ / ZOO / NES …）、各种模型后端（PyTorch / ONNX / TensorRT / Paddle 等）和各种任务类型（分类 / 检测 / 分割 / OCR / 人脸）统一到一套 **Attack Runner** 体系里。

重点不是某个具体算法，而是整个系统怎么设计、怎么扩展、怎么稳定运行。

---

## 1. Attack Runner 的角色与目标

在真实工程环境中，你不会写一个 `pgd_attack.py` 就结束，而是需要一个完整的“攻击流水线”：

- 支持多种攻击算法（白盒 + 黑盒）
- 支持多种任务（img cls / det / seg / ocr / face）
- 支持多种模型后端（Torch / ONNX / TRT / Paddle / MindSpore）
- 支持多 GPU / 多进程 / 大数据集
- 能够 **中断 / 恢复**
- 统一输出结构供前端和报告系统使用

可以把 Attack Runner 理解为：

> “对抗评测系统的调度器 + 控制器”。

---

## 2. 总体架构设计

推荐一个分层架构（逻辑示意）：

    +-----------------------------------------------------------+
    |                 Attack Evaluation Pipeline                |
    +--------------------------+--------------------------------+
    |       Attack Runner      |     Metric / Report Engine     |
    +--------------------------+--------------------------------+
    |       Attack Adapter     |        Model Adapter           |
    +--------------------------+--------------------------------+
    |   (FGSM / PGD / C&W...)  | (PyTorch / ONNX / TensorRT...) |
    +-----------------------------------------------------------+

各层职责概览：

| 模块           | 主要职责                                               |
|----------------|--------------------------------------------------------|
| Model Adapter  | 封装模型后端（Torch / ONNX / TRT …），统一 forward/gradient 接口 |
| Attack Adapter | 封装具体攻击算法，统一 `step()` / `run_on_dataset()` 接口        |
| Attack Runner  | 任务调度、batch 管理、日志、缓存、中断恢复                         |
| Metric Engine  | 各任务指标计算（top1/mAP/mIoU/CER/EER 等）              |
| Report Engine  | 汇总 JSON / 生成文档 / 导出典型样本                           |

---

## 3. Model Adapter：屏蔽不同框架差异

不同模型后端的 API 差异非常大（尤其是梯度获取能力）。  
所以我们首先定义一个统一的“模型适配器”接口。

### 3.1 抽象接口示例

伪代码结构：

    class ModelAdapter:
        def forward(self, x):
            """
            输入张量 x，返回模型输出（logits / boxes / masks / embedding 等）
            """
            raise NotImplementedError

        def gradient(self, x, y):
            """
            对输入 x 求梯度：∂L/∂x
            白盒攻击会调用；黑盒攻击可以不实现或抛异常。
            """
            raise NotImplementedError

### 3.2 PyTorch 后端示例

    class TorchModelAdapter(ModelAdapter):
        def __init__(self, model, loss_fn, device="cuda"):
            self.model = model.to(device)
            self.loss_fn = loss_fn
            self.device = device

        def forward(self, x):
            self.model.eval()
            with torch.no_grad():
                x = x.to(self.device)
                return self.model(x)

        def gradient(self, x, y):
            self.model.eval()
            x = x.clone().detach().to(self.device)
            x.requires_grad_(True)

            logits = self.model(x)
            loss = self.loss_fn(logits, y.to(self.device))

            self.model.zero_grad()
            loss.backward()

            grad = x.grad.detach()
            return grad

### 3.3 ONNX / TensorRT 后端

- 只能 forward，无法直接求梯度  
- `gradient()` 可以：
  - 不实现（raise NotImplementedError）
  - 或用于黑盒攻击时，通过 NES / ZOO 等方式间接估计梯度

模型后端全部被封装进 Model Adapter，Attack Adapter 和 Runner 都只面对统一接口。

## 4. Attack Adapter：统一攻击算法接口

我们希望“挂一个攻击算法”只需要替换一个类，而不改 Runner。

### 4.1 抽象攻击接口

    class AttackMethod:
        def __init__(self, model: ModelAdapter, params: dict):
            self.model = model
            self.params = params

        def step(self, x, y):
            """
            给定当前样本 x、标签 y，返回单步更新后的对抗样本 x_adv
            适用于迭代类攻击（PGD / BIM / C&W 等）。
            """
            raise NotImplementedError

        def run_on_dataset(self, dataloader):
            """
            可选：对一个 dataset 执行完整攻击流程，返回 adv_inputs / labels 等。
            方便像 Runner 这种高级调度逻辑直接使用。
            """
            raise NotImplementedError

### 4.2 白盒 PGD 实现示例

    class PGDAttack(AttackMethod):
        def __init__(self, model, params):
            super().__init__(model, params)
            self.alpha = params.get("alpha", 2/255)
            self.eps = params.get("eps", 8/255)
            self.clamp = params.get("clamp", (0.0, 1.0))
            self.num_steps = params.get("steps", 10)

        def step(self, x, y):
            grad = self.model.gradient(x, y)
            x_adv = x + self.alpha * grad.sign()

            # 限制在 eps-ball 内
            x_adv = torch.clamp(x_adv, x - self.eps, x + self.eps)

            # 再限制在图像取值范围内
            x_adv = torch.clamp(x_adv, self.clamp[0], self.clamp[1])
            return x_adv

        def run_on_dataset(self, dataloader):
            all_adv = []
            all_labels = []

            for x, y in dataloader:
                x_adv = x.clone()
                for _ in range(self.num_steps):
                    x_adv = self.step(x_adv, y)
                all_adv.append(x_adv.detach().cpu())
                all_labels.append(y.detach().cpu())

            adv_inputs = torch.cat(all_adv, dim=0)
            labels = torch.cat(all_labels, dim=0)

            return {
                "adv_inputs": adv_inputs,
                "labels": labels
            }

### 4.3 黑盒攻击（NES / ZOO / HSJ）

对黑盒攻击：

- `step()` 内部不依赖 `model.gradient()`  
- 使用 query-based 方法估计梯度或边界方向，例如：
  - NES：随机噪声估计期望梯度
  - ZOO：有限差分估计每个维度梯度
  - HSJ：沿决策边界逼近法向量

接口保持不变，Runner 无需重写。

## 5. Attack Runner：整体执行逻辑

Runner 负责把：

- Model Adapter
- Attack Adapter
- 数据集加载器（DataLoader）
- Metric Engine
- 存储 & 报告

这些部分拼在一起。

### 5.1 高层伪代码结构

    class AttackRunner:
        def __init__(self, model_adapter, attack_method, metric_engine, config):
            self.model = model_adapter
            self.attack = attack_method
            self.metric_engine = metric_engine
            self.config = config

        def run(self, dataloader):
            for batch_idx, (x, y) in enumerate(dataloader):
                # 1. 生成对抗样本
                adv_result = self.attack.run_on_dataset_single_batch(x, y)

                x_adv = adv_result["adv_inputs"]
                labels = adv_result["labels"]

                # 2. 计算模型对 adv 的预测
                preds_clean = self.model.forward(x)
                preds_adv = self.model.forward(x_adv)

                # 3. 更新指标
                self.metric_engine.update(x, y, preds_clean, preds_adv)

                # 4. 保存典型样本 / 中间结果
                self.save_batch_result(batch_idx, x, x_adv, y, preds_clean, preds_adv)

            # 5. 汇总最终评估结果
            return self.metric_engine.summary()

这里的 `run_on_dataset_single_batch` 可以只是包装 `step()` 的简单逻辑，  
也可以复用之前定义的 `run_on_dataset()` 接口。

## 6. 多任务支持（分类 / 检测 / 分割 / OCR / 人脸）

Attack Runner 不应该直接计算指标，因为不同任务的输出结构完全不同。  
因此我们需要一个 Metric Engine。

典型任务与指标：

| 任务类型             | 模型输出                  | 典型指标                        |
|----------------------|---------------------------|---------------------------------|
| 图像分类             | logits / prob             | Top-1 / Top-5 准确率            |
| 目标检测             | boxes + scores + labels  | mAP50 / mAP75                   |
| 语义分割             | per-pixel logits / mask  | mIoU / pixel accuracy           |
| OCR（文本识别）      | 序列 / token 序列        | CER（字符错误率）/ WER（词错率）|
| 人脸识别（验证模型） | embedding 向量           | ROC-AUC / EER / TAR@FAR         |

### 6.1 Metric Engine 接口

    class MetricEngine:
        def __init__(self, task_type, config):
            self.task_type = task_type
            self.config = config
            # 内部保存各种统计量

        def update(self, x, y, preds_clean, preds_adv):
            """
            根据任务类型，更新对应的统计量。
            """
            raise NotImplementedError

        def summary(self):
            """
            返回一个 dict，包含本次评测的所有指标结果。
            """
            raise NotImplementedError

Attack Runner 不需要知道“如何算 mAP / mIoU / CER / EER”，  
只需要把原始数据交给 Metric Engine 即可。

## 7. Batch 管理与数据加载

为了让攻击评测在大数据集上可用，Batch 管理必须足够灵活。

常见配置项：

    {
      "batch_size": 8,
      "data_load_processes": 4,
      "random_load": false,
      "timeout_limit": 10000,
      "subset_size": 100
    }

含义说明：

- batch_size：每个 batch 多少样本
- data_load_processes：DataLoader 的 worker 数量
- random_load：是否随机抽样
- timeout_limit：单 batch 最长处理时间（毫秒）
- subset_size：仅使用数据集前多少个样本进行评测

### 7.1 推荐实践

- 对于 C&W / DeepFool 等耗时攻击，适当减小 batch_size
- 使用 `subset_size` 控制评测规模，避免一次性对全量数据做重攻击
- 对 DataLoader 启用多进程以提升 IO 性能

---

## 8. GPU 与多卡支持

Attack Runner 应该支持：

- 单 GPU
- 多 GPU（data parallel）
- 不同任务在不同 GPU 上并行运行（多进程调度）

### 8.1 简单的 device 选择示例

    device = "cuda:0" if torch.cuda.is_available() else "cpu"
    model = model.to(device)

Model Adapter 内部可以：

- 持有 device 信息
- 在 forward / gradient 时自动把张量移入对应 device

多卡场景下：

- 可以为每个 GPU 启动一个 Runner 实例，处理不同数据分片
- 或使用 torch.nn.DataParallel / DistributedDataParallel 等框架封装

核心原则是：

> Runner 控制任务粒度，Model Adapter 负责具体 device 操作。

## 9. 结果结构与存档格式

对抗评测结果必须是“结构化”的，方便后端分析和前端展示。

## 10. 中断与恢复（Resume）机制

真实评测中，一个完整任务可能需要几十分钟甚至几小时，因此 **支持中断恢复** 是必须的。

典型机制：

1. 每个 batch 处理完后，写入一个 checkpoint  
2. checkpoint 中记录：
   - 当前批次索引
   - 已完成样本数
   - 当前累计指标统计（Metric Engine 内部状态）
3. Runner 启动时：
   - 检查是否存在 checkpoint
   - 如存在，则从中断处继续

### 10.1 简单示例

    # 伪代码
    if checkpoint.exists():
        state = load_checkpoint()
        start_batch = state["last_batch_idx"] + 1
        metric_engine.load_state(state["metric_state"])
    else:
        start_batch = 0

    for batch_idx, (x, y) in enumerate(dataloader):
        if batch_idx < start_batch:
            continue  # 已处理，跳过

        # ... 正常攻击 + 评估逻辑 ...

        # 更新 checkpoint
        save_checkpoint({
            "last_batch_idx": batch_idx,
            "metric_state": metric_engine.dump_state()
        })

这样，即使中途崩溃 / 重启，也不会丢失已有的评测结果。

## 11. 小结：一个工程化 Attack Runner 的关键特性

一个成熟的 Attack Runner 应该具备：

1. 模块化  
   - Model Adapter / Attack Adapter / Metric Engine / Runner 分层清晰

2. 可扩展  
   - 新增攻击方法时，只需添加一个 Attack 类  
   - 新增任务时，只需扩展 Metric Engine  

3. 可移植  
   - 支持多框架模型（Torch / ONNX / TRT / Paddle 等）

4. 可观测  
   - 每步进度、每个阶段都有日志与中间结果

5. 可恢复  
   - 具备 checkpoint / resume 能力

6. 可集成  
   - 输出结构化 JSON，方便前端 UI 和报告生成（Word/PDF）复用

在这样的架构之上，你可以稳定地挂上前几章提到的各种攻击算法（PGD / C&W / AutoAttack / NES / ZOO / HSJ 等），  
并对分类、检测、分割、OCR、人脸识别等任务进行统一、可比较的鲁棒性评测。

这一章完成了从“算法层”到“工程系统层”的过渡，为后续构建完整的对抗鲁棒性评测平台打下基础。
